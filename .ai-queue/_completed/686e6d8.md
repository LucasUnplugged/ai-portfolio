## Instructions

### 1. Install additional shadcn/ui components

Run these in the worktree root:
```bash
npx shadcn@latest add input table scroll-area tooltip progress select
```

These are needed by the Ledger demo screens (forms, data tables, sidebar scrolling, tooltips, progress bars, filter dropdowns).

### 2. Create mock data modules

Create `src/data/ledger/` directory with these files. All data should be realistic and deterministic (no random generation). Use professional project management domain language.

**`src/data/ledger/users.ts`** ‚Äî 6 team members:
- Each with: id, name, email, role (PM, Senior Engineer, Designer, Engineer, QA Lead, Tech Lead), avatarInitials (2 chars), status ('online' | 'away' | 'offline')
- Export as `users` array and `getUserById` helper

**`src/data/ledger/projects.ts`** ‚Äî 4 projects:
- Each with: id, name, description, status ('active' | 'archived'), memberIds (references users), documentCount, taskCount, lastUpdated (ISO date string), color (for accent)
- Projects: "Brand Redesign", "API v3 Migration", "Mobile App Launch", "Security Audit"
- Export as `projects` array and `getProjectById` helper

**`src/data/ledger/documents.ts`** ‚Äî 6-8 documents:
- Each with: id, projectId, title, lastEditedBy (userId), lastEditedAt, status ('draft' | 'published' | 'archived'), contentBlocks (array of {type, content} ‚Äî headings, paragraphs, lists, code blocks)
- Realistic document titles like "Technical Architecture", "Sprint Planning Notes", "API Endpoints Reference"
- contentBlocks should have varied content ‚Äî some with 3-4 blocks, some with 8-10
- Export as `documents` array and `getDocumentById`, `getDocumentsByProject` helpers

**`src/data/ledger/tasks.ts`** ‚Äî 15-20 tasks:
- Each with: id, projectId, title, description, assigneeId, status ('backlog' | 'in-progress' | 'in-review' | 'done'), priority ('low' | 'medium' | 'high' | 'urgent'), labels (string[]), dueDate, createdAt
- Distribute across projects and statuses realistically
- Export as `tasks` array, `getTasksByProject`, `getTasksByStatus` helpers

**`src/data/ledger/changes.ts`** ‚Äî 5-6 change entries:
- Each with: id, documentId, authorId, createdAt, status ('pending' | 'approved' | 'rejected'), title, beforeText, afterText (for diff display), comments (array of {authorId, text, createdAt})
- Export as `changes` array and `getChangeById` helper

**`src/data/ledger/conflicts.ts`** ‚Äî 2-3 conflicts:
- Each with: id, documentId, userAId, userBId, sectionTitle, versionA (text), versionB (text), merged (text | null), status ('unresolved' | 'resolved'), createdAt
- Export as `conflicts` array and `getConflictById` helper

**`src/data/ledger/audit.ts`** ‚Äî 20-25 audit log entries:
- Each with: id, userId, action ('created' | 'edited' | 'deleted' | 'commented' | 'approved' | 'rejected' | 'merged' | 'assigned'), targetType ('document' | 'task' | 'project' | 'change'), targetId, targetName, details (string), createdAt
- Chronologically ordered (most recent first)
- Export as `auditLog` array

**`src/data/ledger/index.ts`** ‚Äî Barrel export re-exporting everything from all modules above.

### 3. Type definitions

Create `src/data/ledger/types.ts` with TypeScript interfaces for all data models (User, Project, Document, ContentBlock, Task, Change, ChangeComment, Conflict, AuditEntry). All other data files import types from here.
